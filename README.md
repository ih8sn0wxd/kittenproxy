# 🐱 KittenProxy - WebRTC-Based tunnel via VK voice calls

KittenProxy - Автоматизированный прокси-сервер для проброса вашего интернет-траффика через TURN серверы ВК. 


# Принцип работы
Как было уже написано выше - KittenProxy работает через TURN серверы ВК. Весь интернет-траффик исходящий из вашего устройства передается в зашифрованном виде (DTLS 1.2 + AES-256-GCM). Это означает, что ключ для декрипта вашего траффика лежит только у вас, и прочитать его при отсутствии AES-ключа невозможно.

Сервер выполняет следующую задачу: при запуске автоматически начинает пуллить VK API на новые события аккаунта. Если получает событие на входящий звонок - вступает в него, и ждет подтверждение от клиента.

Клиент же в свою очередь как только включается звонит на аккаунт сервера.

После успешного установления соединения, клиент открывает SOCKS5 Server, куда и должен поступать весь исходящий траффик.

> [!WARNING]  
> Проект находится в разработке, возможны различные баги


## Устройство протокола

При установлении успешного соединения клиент отправляет ``ping`` пакет, и ожидает ``pong`` пакет от сервера. Если оба пакета пришли, начинается проброс траффика в следующем виде:

```
  ┌─────────────────────────────────────────────────────────┐
  │                         packet                          │
  ├──────────┬───────────────────┬──────────────────────────┤
  │  1 byte  │      4 bytes      │     0 - 65531 bytes      │
  │   тип    │     stream_id     │         payload          │
  └──────────┴───────────────────┴──────────────────────────┘

  возможные типы в type:
    0x01 CONNECT     client -> server    "open connection"
    0x02 CONNECTED   server -> client    "ready"
    0x03 DATA        bidirectional        raw bytes
    0x04 CLOSE       bidirectional       "end stream"
    0x05 ERROR       server → client     "connection failed"

  ┌─────────────────────────────────────────────────────────┐
  │                 connect payload (0x01)                  │
  ├──────────┬──────────────────────────────┬───────────────┤
  │  1 byte  │            address           │    2 bytes    │
  │ addr_type│                              │     port      │
  └──────────┴──────────────────────────────┴───────────────┘

    addr_type:
      0x01  ipv4    [4 bytes]
      0x03  domain  [1 byte len][string]
      0x04  ipv6    [16 bytes]

  архитектура:
    клиент                          сервер
      │                               │
      │──── connect (host:port) ─────>│
      │                               │── tcp connect ──> target
      │<─── connected ─────────────── │
      │                               │
      │──── data ────────────────────>│──── write ──────> target
      │<─── data ─────────────────────│<─── read ───────  target
      │           something?          │
      │──── close ───────────────────>│── tcp close ────> target
      │                               │
```

Клиент и сервер периодически отправляют ``ping``/``pong`` пакеты. Если один из участников соединения не ответил с таймаутом 5 секунд - соединение считается закрытым.

## Нытье

Все прекрасно понимают, что сейчас происходит в рф. Без прокси/впн в интернет зайти = фактически сидеть в том же БС, и никто не знает сколько еще проживут такие "обходы" белых списков (ркн прекрасно знает про них, и их прекрасно видно, потому-что трудно не заметить кучу висящих соединений 	на один сервер. Рано или поздно яндексу/ вк просто запретят выдавать сервера с белыми айпишниками кому попало).

Реализация KittenProxy напрямую завязана на ВК, который доступен в БС, а это значит даже в наше время во время шатдаунов такая реализация будет работать не хуже чем всем известный VPN.

Пока работают всякие "анти-глушилки" не стоит использовать KittenProxy как основной способ вылезти в нормальный интернет во время шатдаунов ``(но никто не запрещает, если не хотите кормить бездарей aka владельцев всяких впн сервисов).`` KittenProxy - это фан-проект и способ показать, что есть и другие пути по которым можно спокойно пойти, если способ с дедиками у которых белый айпишник прикроют.

## А прикроют ли это?

Скорее да, чем нет, но не полностью. И костыли будет делать ВК, а не РКН. Учитывая стремление ркн запретить всё и вся, будут пинать вк и просить делать хотя-бы что-то. Но это самая обычная игра в кошки-мышки как это происходит сейчас.

Насчет ТСПУ - сразу мимо, траффик полностью зашифрован и без ключа никак не посмотреть куда вы стучитесь (для остальных траффик выглядит как обычный рандомный набор байтов вообще без намека на соединение куда-либо), а ключ есть только на вашем сервере и у вас на клиенте.

# Установка сервера

Для сервера необходимо купить любой 100-мегабитный дедик, и аккаунт вк.
Дальше по инструкции:
```
$ git clone https://github.com/ih8sn0wxd/kittenproxy
cd kittenproxy
pip install -r requirements.txt --break-system-packages
```

Перед запуском необходимо настроить ``config.py``:
- В ``VK_COOKIES`` вставьте куки вашего аккаунта для сервера в виде Netscape
- В ``SERVER_CALL_AUTH_TOKEN`` вставьте ключ авторизации для звонков (как его достать - клик)

Дальше:
```
$ python main.py --mode server
```

Готово. Сервер теперь ждет звонка от клиента

# Установка клиента

Клиент есть на все популярные платформы. Обычный Happ/etc использовать не получится, потому что это не впн. Ссылки на клиенты для Android/iOS будут снизу.

Вам так-же понадобится еще один аккаунт для клиента.

### Использование на Linux/Windows/macOS

Точно так-же скачиваем себе исходники как описано в установке сервера. В конфиге:
- Меняем ``CLIENT_CALL_AUTH_TOKEN`` на ключ авторизации для звонков (как его достать - клик)
- Меняем ``TARGET_BOT_USER_ID`` на айдишник аккаунта на котором стоит сервер.

Опционально: Можно изменить хост/порт в ``PROXY_CONF`` (по дефолту прокся стоит на ``socks5://127.0.0.1:1337``)

```
$ python main.py --mode client
```

Готово. Смотрим логи, если сервер ответил на звонок и прокся стартанула - ставим её в браузер и сидим спокойно в интернете
